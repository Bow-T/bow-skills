# Frontend UI Engineering

## What good looks like

Ship interfaces that feel built by an engineer who understands design: they follow a real design system, handle every state, work for keyboard and screen-reader users, and carry none of the tell-tale "generated by a model" look. The target is UI a senior engineer at a strong product team would sign off on.

This applies whether the surface is a TypeScript/React web app or a Flutter app — the principles are the same even where the syntax differs.

## When to reach for this

- Building a new component, screen, or page.
- Changing an existing user-facing surface.
- Laying out responsive content.
- Wiring up interactivity or client-side state.
- Fixing a visual or UX defect.

## Component structure

### Keep related files together

Co-locate everything a component owns.

React/TypeScript:

```
src/components/TaskList/
  TaskList.tsx          # the component
  TaskList.test.tsx     # its tests
  useTaskList.ts        # extracted hook, if state is non-trivial
  types.ts              # component-local types, if needed
```

Flutter:

```
lib/ui/task_list/
  task_list_page.dart       # the screen / widget
  task_list_view_model.dart # state + logic (see flutter-mvvm)
  widgets/                  # small private widgets used only here
```

### Compose, don't over-configure

A component built from smaller pieces stays flexible. A component with a dozen config props becomes a maze.

```tsx
// Prefer: composition reads like the UI it produces
<Card>
  <CardHeader><CardTitle>Tasks</CardTitle></CardHeader>
  <CardBody><TaskList tasks={tasks} /></CardBody>
</Card>

// Avoid: a prop for every decision
<Card title="Tasks" headerSize="lg" bodyPad="md" content={<TaskList tasks={tasks} />} />
```

### One job per component

```tsx
export function TaskItem({ task, onToggle, onDelete }: TaskItemProps) {
  return (
    <li className="flex items-center gap-3 p-3">
      <Checkbox checked={task.done} onChange={() => onToggle(task.id)} />
      <span className={task.done ? 'line-through text-muted' : ''}>{task.title}</span>
      <Button variant="ghost" size="sm" aria-label="Delete task" onClick={() => onDelete(task.id)}>
        <TrashIcon />
      </Button>
    </li>
  );
}
```

### Split data from presentation

The piece that fetches should not be the piece that renders. This keeps the renderer trivially testable and lets you reuse it.

```tsx
// Data owner: decides which state to show
export function TaskListContainer() {
  const { tasks, isLoading, error, refetch } = useTasks();
  if (isLoading) return <TaskListSkeleton />;
  if (error)     return <ErrorState message="Couldn't load tasks" onRetry={refetch} />;
  if (!tasks.length) return <EmptyState message="No tasks yet" onCreate={createTask} />;
  return <TaskList tasks={tasks} />;
}

// Pure renderer: given data, draws it
export function TaskList({ tasks }: { tasks: Task[] }) {
  return (
    <ul role="list" className="divide-y">
      {tasks.map((t) => <TaskItem key={t.id} task={t} />)}
    </ul>
  );
}
```

The same separation in Flutter: a ViewModel exposes loading/error/empty/data state; the widget switches on it and the leaf widgets just paint. See `flutter-mvvm`.

## Client-side state

Pick the lightest tool that solves the problem. Reaching for a global store on day one is the most common over-engineering.

| State kind | Web (React) | Flutter |
|---|---|---|
| Transient UI (open/closed, hover) | `useState` | local widget state / ValueNotifier |
| Shared by a few siblings | lift state up | a shared ViewModel |
| App-wide, read often, written rarely (theme, auth, locale) | Context | inherited/provided ViewModel |
| Filters, pagination, shareable view | URL search params | route arguments / query params |
| Remote data with caching | React Query / SWR | repository + ViewModel (Supabase client) |
| Complex client state across the app | Zustand / Redux | a provider-scoped store |

Don't thread props through more than ~3 layers that don't use them. If you are, introduce a shared owner or restructure the tree.

## Design system, not improvisation

### Avoid the generated look

Machine-generated UI has a recognizable signature. Reject every piece of it.

| Generated tell | Why it's a problem | Do instead |
|---|---|---|
| Purple/indigo on everything | A "safe" default that makes every app look the same | Use the project's actual palette |
| Gradients everywhere | Visual noise that fights most design systems | Flat, or only the gradients the system defines |
| Max rounding on everything | Uniform `rounded-2xl` flattens the hierarchy real designs use | The radius scale the system defines |
| Generic hero blocks | Template layout disconnected from the real content | Layout that follows the content and the user's task |
| Placeholder lorem ipsum | Hides wrapping/overflow problems real copy exposes | Realistic sample content |
| Big padding on everything | Equal generous spacing kills hierarchy and wastes space | A consistent spacing scale |
| Identical card grids | A shortcut that ignores information priority | Layout driven by what matters most |
| Heavy stacked shadows | Depth that competes with content and slows low-end devices | Subtle or no shadow unless the system asks for it |

### Spacing

Stay on the scale. Don't invent values.

```css
/* on the scale */   padding: 1rem;       /* 16px */
/* on the scale */   gap: 0.75rem;        /* 12px */
/* off the scale */  padding: 13px;       /* arbitrary */
/* off the scale */  margin-top: 2.3rem;  /* arbitrary */
```

### Typography

Honor the heading hierarchy; don't skip levels or borrow heading styles for body text.

```
h1   page title (one per page)
h2   section
h3   subsection
body default text
small secondary / helper
```

### Color

- Use semantic tokens (`text-primary`, `bg-surface`, `border-default`), not raw hex.
- Meet contrast: 4.5:1 for body text, 3:1 for large text.
- Never let color be the only signal — pair it with an icon, label, or shape.

## Accessibility (target WCAG 2.1 AA)

### Keyboard

Every interactive thing must work without a mouse. Use real interactive elements so you get this for free.

```tsx
<button onClick={onClick}>Save</button>        // focusable, Enter/Space handled — use this
<div onClick={onClick}>Save</div>              // not focusable, no keyboard — don't

// Only if you truly can't use <button>:
<div role="button" tabIndex={0} onClick={onClick}
     onKeyDown={(e) => { if (e.key === 'Enter') onClick(); if (e.key === ' ') e.preventDefault(); }}
     onKeyUp={(e) => { if (e.key === ' ') onClick(); }}>
  Save
</div>
```

In Flutter, wrap non-standard tappables in `Semantics(button: true, label: ...)` and ensure they're focus-reachable.

### Labels

```tsx
<button aria-label="Close dialog"><XIcon /></button>   // icon-only needs a name
<label htmlFor="email">Email</label>
<input id="email" type="email" />
<input aria-label="Search tasks" type="search" />      // when there's no visible label
```

Flutter equivalent: `Semantics(label: 'Close dialog', child: ...)` around icon-only buttons.

### Focus management

Move focus when the visible context changes (opening a dialog, revealing a panel), and trap it inside modal surfaces.

```tsx
function Dialog({ isOpen, onClose }: DialogProps) {
  const closeRef = useRef<HTMLButtonElement>(null);
  useEffect(() => { if (isOpen) closeRef.current?.focus(); }, [isOpen]);
  return (
    <dialog open={isOpen} aria-modal="true">
      <button ref={closeRef} onClick={onClose}>Close</button>
      {/* content; trap Tab within while open */}
    </dialog>
  );
}
```

### Never show a blank screen

Empty, loading, and error states are part of the component, not an afterthought.

```tsx
if (!tasks.length) {
  return (
    <div role="status" className="text-center py-12">
      <TasksEmptyIcon className="mx-auto h-12 w-12 text-muted" />
      <h3 className="mt-2 text-sm font-medium">No tasks yet</h3>
      <p className="mt-1 text-sm text-muted">Create your first task to get going.</p>
      <Button className="mt-4" onClick={onCreate}>Create task</Button>
    </div>
  );
}
```

## Responsive

Design mobile-first, then layer on larger breakpoints.

```tsx
<div className="grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 gap-4">
```

In Flutter, use `LayoutBuilder` / `MediaQuery` breakpoints rather than fixed widths. Check the layout at roughly 320, 768, 1024, and 1440 px.

## Loading and motion

Use skeletons for content (not spinners), and optimistic updates so interactions feel instant.

```tsx
function TaskListSkeleton() {
  return (
    <div className="space-y-3" aria-busy="true" aria-label="Loading tasks">
      {Array.from({ length: 3 }).map((_, i) => (
        <div key={i} className="h-12 bg-muted animate-pulse rounded" />
      ))}
    </div>
  );
}

// Optimistic toggle: update the UI immediately, roll back on failure
function useToggleTask() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: toggleTask,
    onMutate: async (taskId) => {
      await qc.cancelQueries({ queryKey: ['tasks'] });
      const prev = qc.getQueryData<Task[]>(['tasks']);
      qc.setQueryData<Task[]>(['tasks'], (old = []) =>
        old.map((t) => (t.id === taskId ? { ...t, done: !t.done } : t)));
      return { prev };
    },
    onError: (_e, _id, ctx) => qc.setQueryData(['tasks'], ctx?.prev),
    onSettled: () => qc.invalidateQueries({ queryKey: ['tasks'] }),
  });
}
```

## Runtime check

Tests cover logic; they don't catch a misaligned layout or a broken focus ring. For web UI, drive the running app in a real browser (Chrome DevTools MCP) to confirm rendering, keyboard order, and console cleanliness before calling it done. For Flutter, run it on a simulator/device.

## Excuses that aren't valid

| Excuse | Reality |
|---|---|
| "Accessibility is a nice-to-have." | It's a legal requirement in many places and a baseline quality bar everywhere. |
| "We'll make it responsive later." | Retrofitting responsiveness costs several times more than building it in. |
| "Design isn't final, so I'll skip styling." | Use the design-system defaults. Unstyled UI reads as broken to every reviewer. |
| "It's just a prototype." | Prototypes ship. Lay the foundation correctly. |
| "The generated look is fine for now." | It signals low quality. Use the real design system from the first commit. |

## Red flags

- A component over ~200 lines — split it.
- Inline styles or arbitrary pixel values off the scale.
- Missing loading, error, or empty states.
- No keyboard pass (try Tab-ing the whole surface).
- Color as the only state indicator (red/green with no text or icon).
- The generated look: purple gradients, oversized cards, stock hero blocks.

## Verification

- [ ] Renders with no console errors/warnings.
- [ ] Every interactive element is reachable and operable by keyboard.
- [ ] A screen reader conveys the structure and content (labels present).
- [ ] Layout holds at 320, 768, 1024, and 1440 px.
- [ ] Loading, error, and empty states are all handled.
- [ ] Spacing, color, and type follow the project's design system.
- [ ] No accessibility violations (e.g. axe-core on web).
