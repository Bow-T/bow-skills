# Idempotency & Exactly-Once

"Exactly-once delivery" does not exist over a network. What you build is **at-least-once delivery + idempotent processing**. The only reliable defense against duplicates is the *receiver* deduplicating. Design for that.

## Step 1 — Classify the operation

Pick the cheapest mechanism that covers your case:

| Operation shape | Mechanism |
| --- | --- |
| Naturally idempotent (`SET status='paid'`) | Nothing — it already is |
| Insert with a known natural key (one row per order) | Unique constraint |
| External side effect with no natural key (charge a card, send email) | Idempotency key + stored result |
| Multi-step / consumer over a stream | Dedup table keyed by message id, inside the txn |

Red flag: "we retry on timeout" with no dedup anywhere. A timeout means *unknown outcome*, not *failure* — the first attempt may have succeeded.

## Step 2 — Get an idempotency key from the caller

The key must be generated by the **client** before the first attempt and reused on every retry. Never generate it server-side — a server-generated key changes per request and defeats the purpose.

```dart
// Flutter: stable key per logical action, NOT per HTTP attempt.
final idempotencyKey = const Uuid().v4(); // created once when user taps "Pay"
Future<Receipt> pay() => _retry(() => api.charge(
      amount: 4200,
      headers: {'Idempotency-Key': idempotencyKey}, // same on every retry
    ));
```

Scope the key correctly: a key is only meaningful for one logical request. Bind it to the user/tenant + endpoint so one tenant's key can't shadow another's.

## Step 3 — Make the dedup atomic

The fatal bug is check-then-act:

```ts
// WRONG: race window between SELECT and INSERT — two retries both pass the check.
const seen = await db.query('select 1 from charges where key=$1', [key]);
if (seen.rows.length) return cached;
await chargeCard(); // both requests reach here
```

Use the database's atomicity. `INSERT ... ON CONFLICT` is your primary tool — the unique index does the locking for you.

```sql
create table idempotency_keys (
  key          text primary key,
  request_hash text not null,           -- detect key reuse with different payload
  status       text not null default 'in_progress', -- in_progress | done
  response     jsonb,
  created_at   timestamptz not null default now()
);
```

```ts
// Atomic claim. Exactly one caller wins the insert.
const claim = await sql/*sql*/`
  insert into idempotency_keys (key, request_hash, status)
  values (${key}, ${hash}, 'in_progress')
  on conflict (key) do nothing
  returning key`;

if (claim.length === 0) {
  const row = (await sql`select * from idempotency_keys where key=${key}`)[0];
  if (row.request_hash !== hash) throw httpError(422, 'key reused with different body');
  if (row.status === 'done')     return row.response;        // replay cached result
  throw httpError(409, 'request in progress, retry shortly'); // concurrent attempt
}
// We own the key — perform the effect, then persist the result.
```

## Step 4 — Persist the result, not just the fact

A bare "I saw this key" is not enough. The retry needs the **same response** the first attempt returned, or the client sees a 409 forever. Store the response body and replay it.

For external side effects, also pass the idempotency key *downstream* if the provider supports it — most payment APIs accept an idempotency key header. Then your dedup and theirs reinforce each other.

## Step 5 — Bind the effect and the bookkeeping in one transaction

The side effect and the dedup record must commit together, or you reintroduce the duplicate.

```ts
await db.transaction(async (tx) => {
  await tx`update wallets set balance = balance - ${amount} where id=${id}`;
  await tx`insert into ledger (key, amount) values (${key}, ${-amount})`;
  await tx`update idempotency_keys set status='done', response=${json}
           where key=${key}`;
});
```

If the effect is a **non-transactional external call** (charge an external API), you can't put it in the DB transaction. Use the outbox/saga shape:
1. In one txn: claim key (`in_progress`) + write an `outbox` intent row.
2. A worker reads outbox, calls the external API *with the same idempotency key*, then marks `done`.
3. Crash between call and mark → worker retries with the same key → external API dedups.

## Step 6 — Make consumers idempotent (queues / webhooks)

At-least-once delivery is the default for SQS-style queues, webhooks, and `LISTEN/NOTIFY` fan-out. Dedup on the **provider's message/event id**, inside the handling transaction.

```ts
// Webhook handler — event.id is the provider's stable id.
await db.transaction(async (tx) => {
  const ins = await tx`insert into processed_events (event_id) values (${event.id})
                       on conflict do nothing returning event_id`;
  if (ins.length === 0) return; // already handled — ack and move on
  await applyEffect(tx, event);
});
return new Response('ok', { status: 200 }); // only ack after commit
```

Rules for consumers:
- **Ack only after the transaction commits.** Acking first turns a crash into silent data loss.
- **Verify the signature** before trusting `event.id`. Anyone can POST a replay.
- Events can arrive **out of order**. Don't assume `created` precedes `updated`; reconcile to the latest state, or store a version/sequence and ignore stale ones.

## Step 7 — Expiry & cleanup

Keys live forever only if you let them. Set a retention window (24–72h covers realistic retry storms) and reap old rows. Keep ledger/processed-event rows longer than the client's max retry horizon, or a late duplicate slips through after cleanup.

```sql
delete from idempotency_keys where created_at < now() - interval '3 days';
```

Pick the window deliberately: retention shorter than the longest client/queue retry window = a duplicate gap.

## Red flags checklist

- Retrying a non-idempotent write with no dedup key — a timeout already means "unknown."
- `SELECT` then `INSERT` instead of `INSERT ... ON CONFLICT` — race window.
- Idempotency key generated per HTTP attempt rather than per logical action.
- Same key reused for a *different* payload and silently accepted — return 422.
- Acking a message/webhook before the work commits.
- Trusting `event.id` before verifying the signature.
- Side effect committed in a different transaction than the dedup record.
- `PUT`/`DELETE` assumed idempotent but the handler also appends to an audit log per call.
- Cleanup window shorter than the retry window.

## Testing it

- Fire the same request twice concurrently; assert exactly one effect and identical responses.
- Replay a captured webhook payload N times; assert the row count is unchanged.
- Kill the process between the external call and the `done` update; assert recovery produces no duplicate.
- Reorder two events for the same entity; assert final state is correct.
